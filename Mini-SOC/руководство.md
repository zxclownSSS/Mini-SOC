# Руководство по работе с Mini-SOC стендом на базе Wazuh

## 1. Назначение системы

Mini-SOC стенд предназначен для изучения принципов мониторинга событий информационной безопасности, работы SIEM-систем и создания пользовательских правил обнаружения.

В качестве платформы мониторинга используется Wazuh.

Основные задачи системы:

* Сбор журналов событий.
* Централизованное хранение логов.
* Анализ событий безопасности.
* Обнаружение подозрительной активности.
* Формирование оповещений.
* Визуализация событий через веб-интерфейс.

---

## 2. Архитектура стенда

Стенд состоит из двух виртуальных машин.

### Wazuh Server

Выполняет функции:

* Приём событий от агентов.
* Анализ логов.
* Применение правил обнаружения.
* Формирование алертов.
* Отображение информации в Dashboard.

### Windows10-SOC

Выполняет функции:

* Источник событий.
* Генерация журналов безопасности Windows.
* Передача событий на сервер Wazuh через агент.

---

## 3. Принцип работы

Работа стенда осуществляется по следующей схеме:

```text
Windows 10
      │
      │ События безопасности
      ▼
Wazuh Agent
      │
      │ Передача логов
      ▼
Wazuh Manager
      │
      │ Анализ событий
      ▼
Правила обнаружения
      │
      ▼
Alerts
      │
      ▼
Dashboard
```

---

## 4. Основные компоненты Wazuh

### Wazuh Agent

Устанавливается на контролируемое устройство.

Основные функции:

* Сбор логов.
* Мониторинг файлов.
* Мониторинг безопасности.
* Передача данных на сервер.

В рамках проекта агент установлен на Windows10-SOC.

---

### Wazuh Manager

Центральный компонент системы.

Основные функции:

* Получение событий.
* Нормализация логов.
* Применение правил.
* Формирование алертов.

---

### Wazuh Dashboard

Веб-интерфейс системы.

Позволяет:

* Просматривать события.
* Выполнять поиск.
* Анализировать алерты.
* Отслеживать состояние агентов.

---

## 5. Разделы Dashboard

### Endpoints

Отображает подключённые агенты.

Используется для контроля состояния устройств.

---

### Threat Hunting

Основной раздел анализа событий.

Позволяет:

* Просматривать логи.
* Выполнять поиск по событиям.
* Анализировать инциденты.
* Проверять срабатывание правил.

---

## 6. Правила обнаружения

Правила используются для определения событий безопасности.

Каждое правило содержит:

* Уникальный идентификатор (Rule ID).
* Уровень критичности (Level).
* Условия срабатывания.
* Описание события.

Пример:

```xml
<rule id="100003" level="14">
    <if_sid>60170</if_sid>
    <description>
        Custom Alert: User added to Administrarotrs group
    </description>
</rule>
```

---

## 7. Пользовательские правила

Пользовательские правила располагаются в файле:

```text
/var/ossec/etc/rules/local_rules.xml
```

Данный файл используется для создания собственных механизмов обнаружения.

---

## 8. Добавление нового правила

Открыть файл:

```bash
sudo nano /var/ossec/etc/rules/local_rules.xml
```

Добавить правило:

```xml
<rule id="100010" level="10">
    <if_sid>60109</if_sid>
    <description>
        Custom Alert: User account created
    </description>
</rule>
```

Сохранить изменения.

---

## 9. Проверка конфигурации

После изменения правил необходимо проверить конфигурацию.

Выполнить:

```bash
sudo /var/ossec/bin/wazuh-analysisd -t
```

Если ошибок нет, будет выведено сообщение:

```text
Configuration OK
```

---

## 10. Применение изменений

После изменения правил необходимо перезапустить Wazuh Manager.

Выполнить:

```bash
sudo systemctl restart wazuh-manager
```

---

## 11. Проверка срабатывания правил

После создания правила необходимо выполнить действие, которое должно вызвать событие.

Например:

```cmd
net user eviluser Test123! /add
```

После этого открыть раздел:

```text
Threat Hunting
```

и убедиться в появлении нового события.
